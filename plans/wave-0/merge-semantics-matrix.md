# 26 §7 deliverable — Merge-Semantics Matrix

**Scope:** the 8 implemented fold-consumed events + `availability` / `table_state` / `shift_cash` (declared in `FOLDS.md`, unimplemented). Verified against `packages/domain/src/registry.ts`, `states.ts`, `packages/sync-client/src/folds/replay.ts`, `FOLDS.md`, and specs 01–26.

**Conventions used throughout:**
- **G-Set / G-Map** = grow-only set/map, join = union. **UKS** = unique-keyed sum (G-Map keyed by an idempotency token, projected as Σ). **MVR** = multi-value register (retain all, no tiebreak). **Head-set** = `heads(S) = { e ∈ S : e.id ∉ ⋃ b.supersedes }` over a supersedes-DAG. **Monotone OR** = non-emptiness of a G-Set.
- `payloadHash(e) = sha256(canonicalJson(e.payload))` — the platform's **clock-neutral** tiebreak primitive. `min(envelope.id)` is **banned as a value tiebreak**: `00 §6` pins ids to UUIDv7, whose leading 48 bits are the minting device's wall clock, and `envelope.ts:6` validates only `z.string().min(1)` — so id-min is min-wall-clock with no validator behind it.
- **Every convergence claim below is scoped to `(same delivered set, same retention scope, same slice)`.** The observed set is *not* unconditionally grow-only: `01-F41` halts delivery, `01-F42` purges, `01 §5` prunes. Shrink is an **outer-layer key-set operation** (`retentionDrop(keys)` → scoped rebuild), never an inverse merge — no semilattice has one.

**The list is not empty.** It has **four entries, and only one of them touches a device fold.** Everything else in all six folds converges with zero ordering metadata — no `global_seq` read, no `lamport_seq` read, no clock read — which is exactly 26 §3's stated target ("zero fold work on adoption"), satisfied structurally rather than by optimisation.

| **1** | **01-F29: naming *which* refund busted the un-refunded remainder** | Gateway `global_seq` assigned at merge (01-F3), per **DEC-SYNC-007** (accepted) | The *set* predicate `Σ refunds > cap` is order-free and converges everywhere. ### Things that look like ordering problems and are not — the instrument is different in each case

Stating these explicitly, because conflating "order" with "time", "bucket" and "completeness" is what produced most of the defects this matrix corrects:

| Apparent ordering problem | What it actually needs | Why an order would not fix it |
|---|---|---|
| Shift / day / drawer bucketing of a payment | A **carried key** | Ordering the events does not decide which bucket they fall in when the *boundaries* are themselves untrusted stamps from two other devices. |
| `shifts.open_at`, `day.business_date`, `confirmed_at` | A **time source** (DEC-TIME-001) | A total order tells you sequence, not instant. These fields want an instant. |
| 01-F30 conservation | A **closure** mechanism (Auditor over the merged log, DEC-TEST-003) | Both sides are non-monotone in the observed set; the problem is *missing data*, not *disputed data*. |
| `orders.settled` | An **event** (`order.settlement_closed`) | A derived predicate over an append-only log cannot be both monotone and a pure function of the set. Adding an order does not dissolve that. |
| Duplicate shift/day open, table anchor chain | A **carried causal link** (`prev_shift_id`, `supersedes[]`) | A sequence cannot distinguish a legitimate 3-move chain from a 3-way concurrent conflict; the emitter's own observation can. |
| Over/short, COD due | A **carried fact** | No sequence tells a device about a payment sitting in another device's outbox. |

## 4. The three prototypes

**The acceptance oracle changes shape (26 §7):** each prototype asserts **fold-specific convergence** — *equal delivered set ⇒ equal projected state*, over arbitrary delivery orders, duplications and subsets — **not** equality to one universal canonical replay. The current `refold-equivalence` gate must not be reused as-is; it encodes the comparator being removed.

### Prototype A — payment / refund totals

**Build:** `pay[order][attempt] → {event_ids, amount, method}` and `refund[order][attempt]`, plus `refundsOf[payment][attempt]` for the 01-F29 cap only. Materialized maps, not scalars. Requires `payment.refunded.{order_id, settlement_attempt_id}` and the `purpose` discriminator.

**Acceptance predicate:**
1. **Convergence:** ∀ permutations π and duplication multisets of a delivered set S, `project(fold(π(S))) == project(fold(S))` **byte-identically** (canonical JSON, UTF-16 key sort).
2. **Subset soundness:** ∀ S′ ⊆ S with every attempt key single-valued, `pay_total(S′) ≤ pay_total(S)` — an exact lower bound. **Explicitly NOT asserted in the malformed case:** a divergent second member moves a key to zero, so `pay_total` **can decrease** on delivery. Any test asserting blanket monotonicity is asserting a falsehood.
3. **Idempotence under intent, not just transport:** two events with **distinct envelope ids** and the **same** `settlement_attempt_id` contribute **once**.
4. **Divergence:** two events, same attempt id, **different** amounts ⇒ key marked `amount_disputed`, contributes **0**, order raises `attempt_amount_divergence`, and the fold **does not pick**.
5. **Khata:** a `repays_receivable` payment does **not** increment `pay_total`.
6. `settled` is unaffected by every one of the above.

**Counterexamples it must survive:**
- **[P0, min-by-id]** Order O billed 185,000. C1 (correct clock) emits `{attempt:K, 185000}`, id `07f2…`. C2 (clock 6 months behind) flushes a stale retry `{attempt:K, 500}`, id `01a9…`. Under min-by-envelope-id the *backdated* device wins and `pay_total` **drops 185,000 → 500** on delivery. **Required outcome:** key disputed, contributes 0, both members retained, quarantine at the gateway, no tiebreak.
- **[P0, khata]** Day 1 khata settlement `{K1, 185000, khata_credit}`; Day 4 cash repayment `{K2, 185000, cash}`, same order (02-F14). Naive UKS → `pay_total = 370000` → "OVERPAID ₨1,850 — refund to close" on every device, and the prescribed remedy puts ₨1,850 of real cash out of the drawer. **Required outcome:** `pay_total` unchanged by the repayment; receivable decrements.
- **[P1, misattributed refund]** `payment.refunded{order_id:"47", payment_id:"P52"}` where P52 lives on a cloud-stranded LAN-only terminal. The gateway cross-check **never runs**. Assert the fold still converges and that the annotation never changes the integer — and record (§6) that the check is unreachable.
- **[P1, tendered]** `{amount_paisa: 200000}` on a ₨1,850 bill (change ₨1,500 handed over). Assert the fold does **not** offer a refund affordance.

### Prototype B — concurrent table assignment

**Build:** supersedes-DAG head-set for `order.table_assigned` (+ the `order.created` root node), the inverse index `orders_at(T)`, `table_ids` / `table_conflict` / `table_conflict_visibility`, and the **entitlement-union routing predicate** in a simulated hub.

**Acceptance predicate:**
1. **Convergence:** equal delivered sets ⇒ equal `table_ids` (canonical-JSON array, UTF-16 sorted **on the table_id value**) and equal `table_conflict`, over all permutations and duplications.
2. **Chain vs conflict:** a 3-move chain T7→T12→T4 with correct `supersedes` yields `|heads| = 1` (T4). Three concurrent moves yield `|heads| = 3`. **These two must be distinguishable** — this is the whole test.
3. **No clock, no sequence, no id read** anywhere in the head computation or in the routing predicate. Assert by construction: injecting arbitrary `global_seq`, `device_created_at` and `envelope.id` perturbations changes nothing.
4. **Value-equality auto-clear:** two heads naming the **same** table ⇒ **no** conflict.
5. **Shrink:** dropping a superseding event **must not** resurrect a retired head — the tombstone set is materialized.
6. **Routing:** a device entitled to *either* head table receives *all* of that order's assignment events, so it computes the complete head set.

**Counterexamples it must survive:**
- **[P0, delivery-layer arbitration]** Waiter W owns T1–T5, X owns T6–T10. O created by W at T1 (events 4001–4004). At 20:10 the counter emits `{O → T7}`, gseq 4310. X's cursor is at 4309, and **01-F9 is forward-only**, so 4001–4004 are never re-offered: X's handheld shows T7 occupied with a **blank bill, permanently**. Now add concurrency: W moves to T3, counter moves to T7. The hub must emit one boolean per event per device against a value that has no single value; every naive resolution (deliver to both / neither / min-id) is a silent arbitration by sync metadata — and min-id is min-wall-clock, so **a sub-second phone-clock skew decides which waiter is shown a live order.** Required: union routing + key-complete backfill.
- **[P0, slice-blind silence]** W3 owns T1–T8. Counter emits `{O→T12}`, manager emits `{O→T4}`. W3 is denied the T12 event by 01-F40 and computes `|heads| = 1` → renders "Table 4", **no badge**, sync indicator green. 04-F12 requires the badge on *every* surface showing the table. Required: union routing removes it for this key family; `conflict_visibility` marks the residual.
- **[P1, retention]** Prune the assignment root of an order parked overnight (02-F11): heads = ∅ → the order drops off every floor map while its bill is open. Required: materialized heads + atomic per-entity prune.

### Prototype C — competing line-state transitions

**Build:** per-line edge G-Set with `line_context`, `heads()` by set difference over `preds`, ≼-max over the non-terminal total chain, terminal absorption, the contested MVR, the adoption clause, and the **split cooking-done / billable predicates**.

**Acceptance predicate:**
1. **Convergence:** equal edge sets ⇒ equal projected state (single value **or** the contested set, rendered in `ORDER_LINE_STATES` index order) and equal anomaly map, over all permutations, duplications and re-deliveries.
2. **Non-terminal join:** ≼-max over the total chain is monotone — adding any edge never decreases the value. No tie rule is invoked, ever.
3. **Terminal contest:** two distinct grounded terminals ⇒ the projected state **is the set**, byte-identical on both devices, **invariant across the `global_seq` assignment boundary** (assert by folding once before and once after assigning sequences).
4. **Adoption:** a resolution edge naming both heads collapses `|heads|` to 1. Two differing resolutions ⇒ contested again with two heads — the algebra reports that two humans disagreed rather than diverging.
5. **`illegal_transition` is never recomputed away** by any reordering — it is a function of one event's own payload.
6. **Kitchen decoupling:** a contested line counts as **exited** for `lines_ready` and 03-F17, so the ticket clears. `settled` is **not** affected by contest.
7. **Slice-shrink:** losing a line key drops the key **wholesale**; the device asserts nothing about it rather than a fragment.

**Counterexamples it must survive:**
- **[P0, ack-boundary flip]** Concurrent `→voided` and `→served` on a paid line. Under the comparator this yields `settled = 1` / nothing owed before ack, then `settled = 0` / 50,000 paisa unmatched after `global_seq` is assigned — **from the identical event set**. Required: both devices report the same contested pair from the moment both edges are visible, and the ack changes nothing.
- **[P0, zombie tickets]** Under "contested = neither ready nor exited", `lines_ready` never reaches `lines_total`, 03-F17 never fires, the card never leaves the pass, and its 03-F25 clock keeps running — so by late service the pass carries several permanently-red zombies and a genuinely late order is visually identical to them, destroying 03-F14 and 03-F23 simultaneously. Required: any terminal head ⇒ done cooking.
- **[P0, settle gating]** C1 and C2 (both `counter_electron`, both full-slice) partition 20:12–20:31. C1 voids L3; C2 serves L3 and settles ₨1,800; guests leave 20:18. At heal, an AND-guard forces `settled → 0` on a paid order whose customer left — and `contested` clears only by a human merge edge, so the order is **permanently wedged** and its table unclearable mid-rush. **This is strictly worse than the comparator it replaces** (which self-heals). Required: contested never gates settle.
- **[P2, grounding liveness]** Two rapid station taps 400 ms apart; the second reads a pre-append head and emits `preds: []` with `from_states: ['in_prep']`. Under an LFP grounding gate this edge grounds on **no device, ever**, with no anomaly class and — since this type no longer parks — no rolling-window backstop. Required: no grounding gate; every legal edge participates in the max.
- **[P2, KDS vocabulary]** Assert **no** sync-mechanism string, actor name, or second colour system reaches any station or pass surface (03-F18, 03-F21, 03-F23).

## 5. Open product decisions (founder, not engineer)

1. **Ratify `order.settlement_closed` as a new 01 §4 event type.** Gates the money, table *and* line domains simultaneously. Cost: one extra emission at close plus a new type. The alternative is amending 01-F33 to permit derived reopening — which the live code already does on every late line-add.
2. **Reconcile 01-F30 with 01-F32 on khata.** As written they contradict: either the khata settlement is not a `payment.recorded` (02-F12 drops it as a method), or a `purpose` discriminator lands and 01-F30's equation is restated over tendering payments only. Until one lands, **the khata tab is structurally always zero and every repayment double-counts.** Blocks doc 17 and any khata UI.
3. **"Keep the change."** A cash tip and a change-inclusive tender are the same integer to the fold; the cashier must record *something* or the 02-F23 count will not balance; DEC-MONEY-004 records tips as unmodeled and the corpus has **no cash-rounding rule at all**. This decides whether an excess-tender state is an exception, and it **blocks the shift-close variance alarm** — shipping 13-F10 before it means firing a 21 §5 **S1** at a named cashier on the base case of a Friday night.
4. **Does a contested line count toward `billed_effective`?** Both choices converge identically; it is money policy, and it must be one constant in `domain`. For: post-KOT removal requires an approved `void.recorded` (02-F20), so the unapproved side should not silently erase revenue. Against: it bills a customer for food someone recorded as voided.
5. **Does the hub-emitted-milestone reversal change 04-F11, or only clarify it?** 04-F11 names the three derivations but never says *where* the side-effect is computed. Reading it as "computed at the hub and emitted" is the only reading that survives 04-F16/F17 slicing — but it makes the **hub a business-event emitter for the first time**, which is an architecture call, not a doc edit.
6. **Simpler alternative worth considering before the payload schema is written:** drop `seated` from the ledger. If "guests sitting, nothing ordered" is device-local ephemeral UI, occupancy identity derives from the order set, `occupancy_id` disappears and the whole epoch DAG goes with it. Cost: the floor map loses `seated` and a host cannot mark a table held.
7. **May a cashier close a shift or the day on a money set the device knows is incomplete?** 09-F18 already blocks day close on undecided rider orders, so a precedent exists — but a blanket block collides with 01-F17 / 05-F8. Note the redesign deliberately removes this question from the **settle** path (settlement is an act, not a proof); it remains live for shift and day close.
8. **Availability `false`-wins or `true`-wins among concurrent heads?** Now a pure product call, since the safety argument that used to decide it is withdrawn. 01-F17 and the 04 §4 availability-race flow both say the product tolerates overselling and apologising rather than blocking. It changes one operator.
9. **Accept the stale-86 residual?** A genuine 86 from a device offline for an hour will grey the item on arrival and cost one manager tap. A recency rule is rejected on principle — "recent" is a clock, and 25 §12 makes every clock arbitrarily wrong, so an expiry rule hands the 86 decision to whichever tablet was flashed last.
10. **What does a KOT header print for an order with two head tables?** The paper structurally requires one number. Printing both, "TABLE CONFLICT", or nothing are all defensible; **printing "the default head" is not**, because there is no clock-free default. The one place "retain both, never silently pick" collides with a physical single-valued output.

## 6. Unresolved critic findings

These are **not** answered by the matrix above. Listed by whether the revision *created* the problem, *acknowledged and could not fix* it, or *deferred* it.

### 6.1 Created by the fix — new, and the sharpest one

**(a) Entitlement-union routing contradicts 04-F17's privacy law.**
The fix for the P0 delivery-layer circularity routes an order's detail to `{creator} ∪ {section owner of any table named by ANY observed assignment node}` — deliberately **grow-only**, because that is what makes it monotone and clock-free. But once an order moves from waiter A's section to waiter B's, it *is* "other waiters' order detail", which 04-F17 excludes **by design (privacy + storage)** and enforces server- and hub-side. So the union either (i) violates 04-F17 permanently for every moved order, or (ii) is trimmed on move — at which point it is no longer monotone and the head set can be computed from a truncated DAG on the losing device, which is the defect it was written to remove.
**Status: genuinely open.** It needs a founder/architecture ruling on whether "an order I once held is still mine to see", or a different construction entirely. Nothing in the matrix resolves it, and the table and order-lines rows both quietly assume (i).

**(b) Hub-as-business-emitter has no failure story and no owner.**
The milestone reversal is correct and unavoidable, but: 01-F13 hub election runs over `{counter_electron, counter_rn, kitchen}`, so a **kitchen tablet may become the authority for table states**. FOLDS.md does not register `table_state` for `kitchen`. During the <10 s re-election window milestones stall — accepted — but no rule says what a device renders in that window, and no spec assigns the emitter role. Flagged as an open question in the matrix; not answered.

### 6.2 Acknowledged and provably unfixable by any algebra

**(c) Availability subset-blindness (critic P1-5/P1-17).** The revision withdrew the false safety claim and added `head_link`, which is honest — but the underlying hazard is **unfixed**: a device that missed an 86 over the LAN fast path computes `true` where the branch computes `false`, shows `head_count = 1`, and fires no alarm. The revision states plainly that "safety in that direction is supplied by delivery, not by algebra". **No ordering mechanism helps** — the problem is missing data. What remains is a delivery-completeness requirement nobody has specced.

**(d) `conflict_visibility` does not make a slice-blind device *right*, only *honest*.** On a scoped device a single head can be a confidently-decided value the branch knows is contested, and **no delivery ever fixes it** because 01-F40 permanently denies the competing event. Union routing removes this for `order:`/`table:` keys; it does **not** remove it for `item:` keys.

**(e) The cross-epoch `confirmed_at` anchor is still clock-biased.** The set-wise basis selection fixes the non-transitivity (a real 01-N1 break in the previous draft), but the `cross_epoch_syntactic` branch falls back to `envelope.id` — and `payloadHash` cannot separate confirms, because `order.confirmed`'s payload is `{order_id}` alone and therefore identical across members. The defence is that 02-F9 makes a multi-member set a partitioned double-accept seconds apart. **That defence fails if genuinely concurrent confirms can be minutes apart**, and no FR bounds it.

### 6.3 Authority inversion — two instances, no render rule

**(f) The designated total-order authority has less data than the devices it is authoritative over.**
DEC-SYNC-007 relocates the 01-F29 cap check to the gateway "because the gateway legitimately has a total order". It does — but while **T-01-12 / DEC-SYNC-009 is unshipped and the code still enforces superseded DEC-SYNC-004**, a LAN-only till is cloud-stranded, so the gateway's observed set is a **strict subset** of a full-slice in-branch device's (01-F14). A gateway `within` is therefore not merely provisional but **systematically weaker** than a device `violated`. Since 18 §6 makes the manager console a *cloud* screen, 05-F20 inherits the truncation.
**No rule says which verdict renders.** The matrix says "must not render gateway `within` as clearance" and stops there. Same shape applies to the gateway-side `payment.refunded.order_id` cross-check (§4 Prototype A counterexample 3) and to the Auditor's 01-F30 conservation check, which currently runs on a permanently truncated log and will report violations that are artifacts of the transport gap.
**This is not a merge-semantics problem and cannot be fixed here.** It is an argument for prioritising T-01-12 above shipping shift close and the money folds.

### 6.4 Deferred, with named cost

**(g) 21 §5 severity consolidation was not actually done.** Four domains each proposed severity tables independently, with no cross-check and no single PR. They already disagree in shape (shift-cash proposes carried-snapshot variance = **S1** while simultaneously gating it on DEC-MONEY-004; table proposes `table_conflict` = S2 on counter / S3 on waiter; order-lines proposes S3 for everything). 21 §5 states that "no module invents its own alarm behavior — new signal types are assigned a severity **here**". **One consolidated 21 §5 PR is required before any of this renders**, and it must carry the binding rule the analysis produced: *no signal computed from a non-monotone quantity may be assigned S1* (otherwise sync latency alone fires the loudest interrupt in the platform, nightly, killing the channel that also carries 03-F5 print failure).

**(h) The snapshot node itself is unspecified.** §2 entry 3 names `global_seq` as the prune-watermark instrument, but nothing specifies how a synthetic snapshot root is minted, agreed, or verified across devices with per-class retention windows (01-F14). Two independent domains reached this same exception, which is evidence it is real and narrow — but it is a design gap, not a design.

**(i) `station_id` does not exist.** 03-F18/F19/F21 require per-station filtering and call the bump "station-scoped", and no station identifier exists in any payload or in the envelope. Flagged, not asserted — it is a doc-03 decision whether it belongs on the event, on the catalog item, or in a read model.

**(j) 01-F10 needs an amendment or an explicit reading.** Three folds now satisfy "never crash or drop" by constructions that never park (key-presence sets, field-wise lattices, unconditional edge storage). 01-F10 describes parking as *the* mechanism. Either amend it, or state that parking is one permitted implementation of out-of-order tolerance rather than the required one.

**(k) `cap_status: unknown` as a permanent resting state is undecided.** A refund whose parent is never delivered (reachable under 01-F40, and today under the T-01-12 gap) sits at `unknown` forever — honest, but it means 01-F29's cap is unenforceable for that refund **by design**. Related and still blocking: DEC-MONEY-006(a) — does a *suppressed* duplicate settlement still need its event id to resolve refunds? Adding `order_id` to `payment.refunded` dissolves the total half but not the cap half.

# Addendum — prototype results (July 2026)

All three §4 prototypes were built and run (scratchpad-only, repo untouched; each imports the real `canonicalJson` / state machine from `domain`). **All green**: A 13/13, B 6/6 (13 named checks, ~2,300 assertions), C 13/13 (13,548 folds). Every P0/P1/P2 counterexample above was constructed exactly and survived. Combined algebra size across all three hard cases: **~470 non-blank lines** (A 233 · B 117 · C ~120 core) — no comparator, no clock, no sequence.

**Strength of evidence.** All three enforce "no ordering metadata" *dynamically*: folds run over Proxy-poisoned envelopes that throw on `global_seq`/`lamport_seq`/`device_created_at`/`device_id` reads (A), and all scenarios re-run under garbage sequence/clock injection plus **consistent bijective id relabeling including an order-reversing one** (A/B/C). All three mutation-tested their own oracles (planted defects go red).

**The oracle lesson (binding for the real test suite).** C's mutation testing showed a min-envelope-id tiebreak **passes plain convergence** — min-id *is* convergent; it is convergent-and-wrong (min-id = min-wall-clock). Only the id-bijection perturbation kills it. So: the real acceptance suite MUST include bijective-relabel + sequence/clock-injection invariance, and the old refold-equivalence gate — which would have blessed min-id — must not be ported.

## Corrections to this matrix from the prototypes

**From A (money):**
- **The 01-F29 cap must resolve refund parents by `settlement_attempt_id`, not envelope id.** As written (`refundsOf[payment=env.id][attempt]`), an intent legitimately duplicated under two envelope ids fragments the cap: ₨3,700 refunds against a ₨1,850 payment and `violated` never fires. Resolving by attempt id deletes the last envelope-id-*semantic* read from the money algebra and dissolves the cap half of DEC-MONEY-006(a).
- **The immutability contract is the whole payload minus its key**, not `(attempt_id, amount, method)` — two members differing only in `purpose` move `pay_total` by the full amount depending on which a fold picked. Any divergence → disputed → 0. (`attempt_amount_divergence` is now a misnomer.)
- **Cross-order attempt-id collision is fold-undetectable in principle** under nested keying — two orders sharing an attempt id never meet. The org-global uniqueness law is enforceable at mint time and gateway only; the fold cannot backstop it.
- Byte-identity claims pin to the **projection** (id-free); fold state legitimately retains `event_ids` for forensics. CE4 ("no refund affordance") is a projection-shape constraint: `billed` never enters the fold, so overpayment is *underivable* — assert the key set, not UI behaviour.
- 02-F14's plural "order(s)" is unrepresentable with a single `order_id`; multi-order khata repayment = N events or a customer-scoped payload. The receivable's true key is `customer:` (absent); order-keyed is a single-order stand-in.

**From B (tables):**
- **Predicate 3 as literally written is unimplementable** — `supersedes` *references* ids, so independent id scrambling severs the DAG for any implementation. The honest invariant: ids are identity-only ⇒ projections invariant under **consistent bijective relabeling** (including order-reversing). Pin "no id read" as "no id **comparison/ordering**"; identity reads (node keying, tombstones, dedupe) are necessary.
- **`conflict_visibility` is NOT monotone.** Under union routing every device is briefly pre-entitlement, so a grow-only `withheld` marker brands every member slice-blind forever. `withheld` clears on key-complete backfill — it is a statement about delivered-vs-full *now*, maintainable only by the layer that did the withholding.
- **Materialized tombstones prevent resurrection but do not close retention** — per-event pruning still yields `heads = ∅` (order off the floor map, bill open). The load-bearing half is **atomic per-entity prune granularity + an open-bill guard**.
- Pin: `from_table_id` **counts** as "named" for the entitlement union (it is what entitles the origin-section owner and covers `table_id: null` creations); `order.created` **is** a legal `supersedes` target (every counterexample requires it); "creator" needs device-vs-user resolution (04-F13 is user-scoped, BYOD blurs it); the union is computed against `owner(T)` — a **mutable** layer-2 section map, so 04-F18 mid-shift reassignment requires re-evaluating owners of already-named tables.

**From C (lines):**
- **The ≼-max must range over ALL legal non-terminal edges, not over heads** — a legal edge can retire a higher head and *lower* a max-over-heads, breaking monotonicity. `heads()` answers only the terminal-contest/adoption question. (The P2 liveness counterexample's "every legal edge participates" is the only reading under which predicate 2 holds.)
- **Rows 63 and 66 contradict** on sequential void-after-serve: the row-63 parenthetical implies preds can retire a served head into voided; row 66 correctly says post-serve void is NOT EXPRESSIBLE under strict legality. Row 66 wins; fix the row-63 text (or deliberately extend adoption to singleton-terminal supersession — a product call, not a default).
- "Grounded terminal" is undefined in a design that forbids grounding gates — define as *payload-legal terminal head* or rename.
- **Cooking-done must include `picked_up`**: 03-F17's exit set is "served or picked_up", and `picked_up` is non-terminal — any-terminal-head alone strands delivery lines on the pass until `delivered`. Exit = any-terminal-head ∨ watermark ≥ `picked_up`.
- `from_states: []` must be `illegal_transition` (∀ over ∅ is vacuously legal) — real schema needs `.min(1)`. Anomaly G-Map needs a stated priority (illegal > inconsistent_predecessor > terminal_regression). `LEGAL_NEXT` needs exporting from `domain/states.ts` (scheduled small change).

## §6.1(a) privacy exposure — measured (the founder number)

After an order moves fully from waiter W's section to X's, under grow-only union routing with order-granular slicing: **W continues to receive 8 events** (1 assignment, 3 `line_added`, 4 `line_state_changed`) including **3 lines of item+price detail = Rs 670**, and **0 payment/cash/shift events** — 04-F17's categorical money exclusion is orthogonal and held. Bounded by the 04-F16 business-day window, not permanent. Reverse direction (required for correctness): X's backfill exposes 7 W-era events = Rs 1,640 of line detail. **The trim alternative was built and shown to re-create the slice-blind defect on the privacy fix itself** (trimmed W projects "T3, no conflict" against branch truth {T3,T9}+conflict). The ruling is therefore concretely: *one business day of order/line-plane visibility on moved orders* vs *a losing device silently deciding conflicts*.
