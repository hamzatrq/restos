### T-01-03  sync-client device store + outbox core (the canonical durable queue, 18 §4) + kill-test seed
- **FRs:** 01-F2 (persist-before-ack — plug-pull/kill safe), 01-F3 device half (`lamport_seq` monotonic, persisted, gap-free; cloud `global_seq` assignment is gateway work), 01-F8 outbox half (drain in lamport order, advance only to acked watermark, idempotent append/ack; the network push itself lands with T-01-05/gateway), 01-F11 (sync status queryable). Constraints, not closed here: 01-F1 (store API exposes no update/delete), 01-F4 (append validates against the domain registry), 18 §4 canonical-queue laws (never-lose, crash-resume, idempotent-drain), 19 §5 write-checkpoint, 20 §2.6 kill-tests.
- **Files touchable:** `packages/sync-client/src/**` (impl session; protected path); `packages/sync-client/src/__acceptance__/**` (test session only); `packages/sync-client/package.json` + lockfile (deps/wiring: better-sqlite3 pinned per 18 §4, vitest, fast-check — wiring may land with the test session so the red run executes).
- **Check:** `pnpm --filter @restos/sync-client test` + `pnpm verify:01` + root `pnpm verify`.
- **DoD rung:** D1 (in-suite kill seed; the nightly randomized harness and physical rig are later 20 §2.6 rungs).
- **API contract (binding for the test session):**
  - `openStore({ path, identity }): DeviceStore` — `path` = SQLite file path or `":memory:"`; `identity = { org_id, branch_id, device_id }`. SQLite WAL via better-sqlite3 (node/Electron adapter; the RN adapter arrives with the RN app — the adapter seam stays internal until a second consumer exists, 24-F23).
  - `store.append(input): EventEnvelope` — `input` = envelope minus `lamport_seq`/`server_received_at`; the store assigns the next gap-free `lamport_seq` (starting at 0) and `server_received_at: null`, validates through the domain registry (unknown type / invalid payload throws, nothing persisted, 01-F4), requires `input` org/branch/device ids to equal the store identity (mismatch throws), and persists durably **before** returning — a returned envelope IS the confirmed event (01-F2). Re-append of an already-stored event `id` returns the stored envelope unchanged and assigns nothing (idempotent — crash-retry safe, 18 §4).
  - `store.nextBatch(max): EventEnvelope[]` — oldest unacked own events, `lamport_seq` order (the push feed, 01-F8).
  - `store.advanceTo(watermark): void` — write-checkpoint (19 §5): acks all events with `lamport_seq ≤ watermark`. Partial acks legal; idempotent; a watermark below the current checkpoint is a no-op; a watermark above `own_high_water` throws `AckBeyondAppendedError` (loud failure, 18 §4 — an impossible ack means protocol corruption).
  - `store.readOwnEvents(fromLamport?): EventEnvelope[]` — acked events remain readable: the events table is the device's ledger copy (01 §5); compaction is later work.
  - `store.status(): { queue_depth, own_high_water, acked_watermark, last_global_seq }` (01-F11) — `own_high_water`/`acked_watermark`/`last_global_seq` are `null` until first append/ack/pull; `store.setLastGlobalSeq(n)` records the pull position (consumed by T-01-05).
  - `store.close(): void`. No API method updates or deletes an event row (01-F1).
  - **Kill-seed law (20 §2.6, D1 form):** events confirmed before an abrupt handle abandon (no `close()`) all survive `openStore` reopen with no lamport gap, and the unacked tail re-drains identically; at `append` return the event is already visible to an independent second handle on the same file (proves synchronous persistence, not buffering). The true SIGKILL child harness is the nightly 20 §2.6 rung — a follow-on harness task, since a bare-node child needs a TS-execution runtime decision that is infra, not oracle.
- **Assumptions stated:** lamport starts at 0 (envelope permits ≥ 0; consistent with T-01-02 watermark-zero tests); acked events retained, not deleted — simpler delete-on-ack alternative rejected because the events table is the ledger copy (01 §5); loud `AckBeyondAppendedError` over silent clamping (18 §4 loud-terminal-failure); `PRAGMA synchronous=FULL` accepted (00 §5.2 plug-pull law outranks write throughput at this volume); store identity fixed at open — one device, one store; multiple handles on one file are legal for reads (WAL), single writer is the usage contract (enforcement not required at v1).
