- **API contract (binding for the test session):**
  - `openStore({ path, identity }): DeviceStore` — `path` = SQLite file path or `":memory:"`; `identity = { org_id, branch_id, device_id }`. SQLite WAL via better-sqlite3 (node/Electron adapter; the RN adapter arrives with the RN app — the adapter seam stays internal until a second consumer exists, 24-F23).
  - `store.append(input): EventEnvelope` — `input` = envelope minus `lamport_seq`/`server_received_at`; the store assigns the next gap-free `lamport_seq` (starting at 0) and `server_received_at: null`, validates through the domain registry (unknown type / invalid payload throws, nothing persisted, 01-F4), requires `input` org/branch/device ids to equal the store identity (mismatch throws), and persists durably **before** returning — a returned envelope IS the confirmed event (01-F2). Re-append of an already-stored event `id` with identical content returns the stored envelope unchanged and assigns nothing (idempotent — crash-retry safe, 18 §4); a re-append whose content diverges from the stored row (any field beyond the two store-assigned ones) throws with nothing changed — silent divergence between caller belief and ledger truth is the 18 §4 loud-failure case (adversarial-review finding).
  - `store.nextBatch(max): EventEnvelope[]` — oldest unacked own events, `lamport_seq` order (the push feed, 01-F8).
  - `store.advanceTo(watermark): void` — write-checkpoint (19 §5): acks all events with `lamport_seq ≤ watermark`. Partial acks legal; idempotent; a watermark below the current checkpoint is a no-op; a watermark above `own_high_water` throws `AckBeyondAppendedError` (loud failure, 18 §4 — an impossible ack means protocol corruption). The watermark must be a non-negative integer: NaN/negative/fractional values throw with the checkpoint unchanged (same corruption class — adversarial-review finding, checkpoint must never silently regress); on an empty store every watermark is beyond high water and throws.
  - `store.readOwnEvents(fromLamport?): EventEnvelope[]` — inclusive from `fromLamport` (default 0), lamport order; acked events remain readable: the events table is the device's ledger copy (01 §5); compaction is later work.
  - `store.status(): { queue_depth, own_high_water, acked_watermark, last_global_seq }` (01-F11) — `own_high_water`/`acked_watermark`/`last_global_seq` are `null` until first append/ack/pull; `store.setLastGlobalSeq(n)` records the pull position (consumed by T-01-05).
  - `store.close(): void`. No API method updates or deletes an event row (01-F1).
  - **Kill-seed law (20 §2.6, D1 form):** events confirmed before an abrupt handle abandon (no `close()`) all survive `openStore` reopen with no lamport gap, and the unacked tail re-drains identically; at `append` return the event is already visible to an independent second handle on the same file (proves synchronous persistence, not buffering). The true SIGKILL child harness is the nightly 20 §2.6 rung — a follow-on harness task, since a bare-node child needs a TS-execution runtime decision that is infra, not oracle.
- **Assumptions stated:** lamport starts at 0 (envelope permits ≥ 0; consistent with T-01-02 watermark-zero tests); acked events retained, not deleted — simpler delete-on-ack alternative rejected because the events table is the ledger copy (01 §5); loud `AckBeyondAppendedError` over silent clamping (18 §4 loud-terminal-failure); `PRAGMA synchronous=FULL` accepted (00 §5.2 plug-pull law outranks write throughput at this volume); store identity fixed at open — one device, one store; multiple handles on one file are legal for reads (WAL), single writer is the usage contract (enforcement not required at v1).

- **API contract (binding for the test session):**
  - **Placement:** folds live in `packages/sync-client` and are always-on — `openStore({ path, identity })` (unchanged signature) creates/maintains the fold tables in the same SQLite file and guarantees, on return, state tables ≡ `refold()` (fold state is durable with the ledger and self-heals on reopen). Apps never register folds (FOLDS.md registration rule, 18 §6).
  - **Canonical replay order (the determinism anchor):** fold state is defined as the fold of the stored event **set** replayed in the total order `key(e) = (global_seq(e) ?? +∞, device_created_at, device_id, lamport_seq)`, lexicographic. Arrival order never matters (01-N1). When every competitor carries a `global_seq`, this is exactly cloud order (01-F34); before assignment the tail key is the deterministic provisional order and every device holding the same set agrees. Incremental maintenance is an implementation choice; the law is equivalence with canonical replay.
  - **New store surface (additive to T-01-03; nothing existing changes shape):**
    - `store.ingest(envelope: EventEnvelopeT, opts?: { global_seq?: int ≥ 0 }): { stored: boolean }` — the branch-stream entry point (mesh/cloud delivery wires in at T-01-05; tests call it directly). Validates via domain `parseEvent` — unknown type/invalid payload throws, nothing persisted (01-F4). `org_id`/`branch_id` must equal the store identity (mismatch throws). An envelope `id` already stored (own or peer) → `{ stored: false }`, zero state change (idempotence). An own-`device_id` envelope not already stored throws (own events enter only via `append` — loud, 18 §4). Persists to the new `peer_events` table (`id PK, device_id, lamport_seq, global_seq NULL, envelope; UNIQUE(device_id, lamport_seq)` — a (device, lamport) collision with a different id throws: lamport is gap-free monotonic per device, so a collision is corruption) and applies folds in the same transaction.
    - `store.append(input)` — unchanged contract from T-01-03, now also applies folds in the same transaction as the insert. Append NEVER fails or blocks for fold reasons (a sale is never blocked, 01-F17): an appended event with an unmet dependency persists to the ledger and parks at the fold layer.
    - `store.assignGlobalSeq(event_id: string, global_seq: int ≥ 0): void` — records cloud ordering for an already-stored own or peer event in a sidecar table (`global_seq` map keyed by event id; envelope rows are never updated, 01-F1). Same value again = idempotent no-op; a different value for the same event, an unknown `event_id`, or a `global_seq` already held by another event throws (protocol corruption — loud, 18 §4). After it returns, state tables ≡ `refold()` (this is how "devices converge to cloud ordering on ack" is observable).
    - `store.openOrders(): OpenOrderRow[]` (sorted by `order_id` asc), `store.kitchenQueue(): KitchenQueueRow[]` (sorted by `order_id` asc), `store.parked(): ParkedRow[]` (sorted by `event_id` asc), `store.refold(): void` (drop + rebuild all fold tables from `events` ∪ `peer_events` + sidecar — the device-side seed of the 20 §4.2 refold-and-diff pattern).
  - **`open_orders` state (SQLite `orders(order_id PK, channel, order_type, table_id, confirmed_at, settled, json_lines)` — exactly FOLDS.md):**
    - `order.created` → row `{ order_id, channel, order_type: payload.order_type ?? NULL, table_id: payload.table_id ?? NULL, confirmed_at: NULL, settled: 0, json_lines: {} }`. A second `order.created` for the same `order_id`: first in canonical order wins, later ones are logged no-ops.
    - `order.confirmed` → `confirmed_at = device_created_at` of the canonically-first confirm; later confirms are logged no-ops.
    - `order.line_added` → line `{ item_id, qty, unit_price_paisa, state: "placed", anomalies: {} }` under `line_id`; duplicate `line_id`: first in canonical order wins. Lines are never removed at v1 (`line_removed` deferred; post-KOT removal is `void.recorded` per 01 §4†).
    - `order.line_state_changed` → for each `line_id` in `line_ids`, `applyLineState(current, payload.state)`; non-applied results write `anomalies[envelope.id] = "terminal_regression" | "illegal_transition"` on that line (retained + flagged, 01-F35; the ledger row is untouched).
    - `order.table_assigned` → `table_id` = payload of the canonically-**last** assignment (the 01-F34 order-sensitive field: once all competitors carry `global_seq`, highest `global_seq` wins on every device).
    - `payment.recorded` / `payment.refunded` → `settled = 1` iff `billed_effective > 0 ∧ Σ payments.amount_paisa − Σ refunds.amount_paisa == billed_effective`, where `billed_effective = Σ qty × unit_price_paisa` over lines with state ∉ {voided, cancelled}, and a refund attributes to the order of its parent payment. `settled` is derived, recomputed deterministically from the set (v1 approximation of 01-F30 — void/comp/discount value terms are 0 until their event types land; refund-remainder enforcement (01-F29) is hub/cloud quarantine work, 01-F37 — both listed above as constraints-not-closed).
    - `json_lines` is canonical JSON (object keys sorted lexicographically at every depth, no insignificant whitespace) — determinism assertions compare it byte-for-byte.
  - **`kitchen_queue` state (SQLite `queue(order_id PK, confirm_at, channel, age_basis, lines_ready, lines_total)` — exactly FOLDS.md):**
    - A row exists iff a canonically-applied `order.confirmed` exists for the order. `confirm_at = device_created_at` of the canonically-first confirm; `channel` from the order's `order.created` (the fold therefore also consumes `order.created`/`order.line_added` — FOLDS.md's consumes list is read as the trigger list; drift flagged for a FOLDS.md correction).
    - `age_basis = device_created_at` of the canonically-first `kot.printed` for the order, else `confirm_at` (kitchen age runs from ticket print when one exists; doc 03 owns timing refinements).
    - `lines_total` = count of lines with state ∉ {voided, cancelled}; `lines_ready` = count with state ∈ {ready, served, picked_up, delivered} (feeds 03-F19/F24's order-ready-is-a-fold).
    - No row exit at v1 — consumers filter; bump/served-exit semantics are doc 03 work.
  - **Parking (01-F10; SQLite `parked(event_id PK, waiting_for, envelope_json)` — exactly FOLDS.md):**
    - Dependencies come from typed payloads (not raw `refs[]`, which stays advisory at v1): every consumed type except `order.created` requires its `order_id` resolved (an applied `order.created` with that payload id); `order.line_state_changed` additionally requires every `line_id` in `line_ids` resolved (an applied `order.line_added`); `payment.refunded` requires `payload.payment_id` resolved (a stored-and-applied `payment.recorded` **event id**).
    - An event with an unmet dependency is persisted to the ledger normally but parks: one `parked` row, `waiting_for` = the first unmet id (deterministic: order id, then lowest missing line id, then payment event id), and it affects NO state table while parked.
    - Drain: when an applied event resolves an id (`order.created` resolves its order id, `order.line_added` its line id, `payment.recorded` its envelope id), parked rows waiting on that id re-attempt in canonical order; still-unmet dependencies re-park under the new `waiting_for`; cascades (refund → payment → order) run to fixpoint. Duplicate ingest of a parked event's id is a no-op (PK).
    - Nothing is ever dropped: every stored envelope of a consumed type is at all times either applied or parked. Rolling-window parked-age anomaly → fleet health is later doc-15 wiring; `store.parked()` is the v1 observability.
  - **Laws (each is an acceptance test; FR id in the test name per 20 §4.1):**
    1. **Determinism (01-N1):** for any registry-valid generated event set (N devices, per-device gap-free lamport) and ANY two delivery interleavings that respect each device's lamport order (own events via `append`, peers via `ingest`), final `orders`/`queue`/`parked` tables are identical, `json_lines` byte-identical (fast-check property; named generator).
    2. **Commutativity (01-F34):** swapping two adjacent deliveries from different devices yields identical final tables (property).
    3. **Idempotence (01-F34):** re-delivering any already-stored events (same ids, any subset, any order) changes nothing — tables byte-identical before/after (property).
    4. **Refold equivalence (01-F6):** after any delivery sequence, `refold()` reproduces the incrementally-maintained tables exactly (property; the Auditor pattern, 20 §4.2).
    5. **Terminal monotonicity (01-F35, regression guard):** a line in a terminal state receiving a later non-terminal `line_state_changed` keeps its state, gains exactly one `anomalies[event_id] = "terminal_regression"` entry, and the ignored event remains readable in the ledger.
    6. **Parking (01-F10):** child-before-parent delivery → parked row present, state tables unaffected; parent arrival → parked row gone, final state identical to parent-first delivery; the refund→payment→order chain delivered fully reversed drains to fixpoint; no consumed event is ever dropped (applied ∪ parked = stored).
    7. **Tiebreak/convergence (01-F34):** two concurrent `order.table_assigned` from different devices with no `global_seq` resolve identically on every delivery order via the provisional key; after `assignGlobalSeq` on both such that cloud order reverses the provisional winner, `table_id` is the higher-`global_seq` value and tables ≡ `refold()`.
    8. **Durability (extends T-01-03's kill-seed):** after abrupt handle abandon (no `close()`), `openStore` on the same path yields state tables ≡ `refold()` of the surviving ledger — confirmed folds are never ahead of or behind the confirmed ledger.

Laws (HUB-ELECTION.md, all property-tested): rank = index in `HUB_ELIGIBLE_CLASSES` (from `@restos/domain`, hub-priority order — lower index wins); tie → lexicographically lowest `device_id`; classes outside `HUB_ELIGIBLE_CLASSES` never win; **permutation-invariant** (`electHub(shuffle(p)) === electHub(p)`) — every device computes the same winner from the same visible set, no consensus rounds.
  - **(c) Mesh session state machine** (`packages/sync-client`): `createMeshSession({ store, transport, clock, device_class, token }): MeshSession` with `start()`, `stop()`, `notifyAppended()` (host-app fast-path trigger: drain outbox now, 01-F15), and `status(): { state: "solo"|"follower"|"candidate"|"hub", hub_id: string|null, peers: PeerInfo[], last_push_ack: number|null }` (01-F11 mesh half).
    - Exported constants (HUB-ELECTION.md, binding): `HEARTBEAT_INTERVAL_MS = 2_000`; `HEARTBEAT_MISSED_LIMIT = 3`; `HUB_LOSS_TIMEOUT_MS = 6_000` (derived); `REELECTION_BUDGET_MS = 10_000` (01-F13 assertion budget — virtual ms here, wall-clock in T-01-06).
    - Transitions: empty peer set → `solo` if hub-eligible (acts as hub for later joiners), else `follower` with `hub_id: null` (waits). Any peer-set change → recompute `electHub(visible ∪ self)`: winner = self → `hub`; winner = other → `follower` (connect: `hello` → `hello_ack{hub:true}`). Hub loss (no wire `ping` for `HUB_LOSS_TIMEOUT_MS`) → eligible devices enter `candidate` (the no-hub recompute window; exits on adopting the new result), non-eligible → `follower`/null. Loss-to-new-hub-connected < `REELECTION_BUDGET_MS` virtual on every device.
    - Hub duties: heartbeat wire `ping{t}` to each connected follower every `HEARTBEAT_INTERVAL_MS` (follower replies `pong`; 3 unanswered → hub drops that follower). On follower `push`: `store.ingestBatch(events)` (persists before ack, 01-F2), reply `push_ack{acked_watermark}`, fan the new events as `event_batch` to every *other* connected follower. On follower `hello`: `hello_ack{ session_id, hub: true, resume_from }` where `resume_from` = next `lamport_seq` the hub expects from that device (highest contiguously held + 1; 0 if none), then replay its stored branch window to the joiner as `event_batch` (id-dedupe absorbs redundancy — LAN catchup does not use `catchup_request`, whose cursor is cloud `global_seq`; see assumptions).
    - Follower duties: on connect and on `notifyAppended`: `store.nextBatch(max)` → `push{events, watermark}` from `resume_from` onward; track `last_push_ack` from `push_ack`. **The hub ack never calls `store.advanceTo`** — the outbox watermark is the *cloud* write-checkpoint (19 §5; HUB-ELECTION.md keeps followers' own cloud fallback); the LAN push cursor is session-local and volatile (worst case re-push; id-dedupe absorbs). Ingest received `event_batch` via `store.ingestBatch`.
