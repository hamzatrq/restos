- **API contract (binding for the test session):**
  - `openStore({ path, identity }): DeviceStore` — `path` = SQLite file path or `":memory:"`; `identity = { org_id, branch_id, device_id }`. SQLite WAL via better-sqlite3 (node/Electron adapter; the RN adapter arrives with the RN app — the adapter seam stays internal until a second consumer exists, 24-F23).
  - `store.append(input): EventEnvelope` — `input` = envelope minus `lamport_seq`/`server_received_at`; the store assigns the next gap-free `lamport_seq` (starting at 0) and `server_received_at: null`, validates through the domain registry (unknown type / invalid payload throws, nothing persisted, 01-F4), requires `input` org/branch/device ids to equal the store identity (mismatch throws), and persists durably **before** returning — a returned envelope IS the confirmed event (01-F2). Re-append of an already-stored event `id` with identical content returns the stored envelope unchanged and assigns nothing (idempotent — crash-retry safe, 18 §4); a re-append whose content diverges from the stored row (any field beyond the two store-assigned ones) throws with nothing changed — silent divergence between caller belief and ledger truth is the 18 §4 loud-failure case (adversarial-review finding).
  - `store.nextBatch(max): EventEnvelope[]` — oldest unacked own events, `lamport_seq` order (the push feed, 01-F8).
  - `store.advanceTo(watermark): void` — write-checkpoint (19 §5): acks all events with `lamport_seq ≤ watermark`. Partial acks legal; idempotent; a watermark below the current checkpoint is a no-op; a watermark above `own_high_water` throws `AckBeyondAppendedError` (loud failure, 18 §4 — an impossible ack means protocol corruption). The watermark must be a non-negative integer: NaN/negative/fractional values throw with the checkpoint unchanged (same corruption class — adversarial-review finding, checkpoint must never silently regress); on an empty store every watermark is beyond high water and throws.
  - `store.readOwnEvents(fromLamport?): EventEnvelope[]` — inclusive from `fromLamport` (default 0), lamport order; acked events remain readable: the events table is the device's ledger copy (01 §5); compaction is later work.
  - `store.status(): { queue_depth, own_high_water, acked_watermark, last_global_seq }` (01-F11) — `own_high_water`/`acked_watermark`/`last_global_seq` are `null` until first append/ack/pull; `store.setLastGlobalSeq(n)` records the pull position (consumed by T-01-05).
  - `store.close(): void`. No API method updates or deletes an event row (01-F1).
  - **Kill-seed law (20 §2.6, D1 form):** events confirmed before an abrupt handle abandon (no `close()`) all survive `openStore` reopen with no lamport gap, and the unacked tail re-drains identically; at `append` return the event is already visible to an independent second handle on the same file (proves synchronous persistence, not buffering). The true SIGKILL child harness is the nightly 20 §2.6 rung — a follow-on harness task, since a bare-node child needs a TS-execution runtime decision that is infra, not oracle.
- **Assumptions stated:** lamport starts at 0 (envelope permits ≥ 0; consistent with T-01-02 watermark-zero tests); acked events retained, not deleted — simpler delete-on-ack alternative rejected because the events table is the ledger copy (01 §5); loud `AckBeyondAppendedError` over silent clamping (18 §4 loud-terminal-failure); `PRAGMA synchronous=FULL` accepted (00 §5.2 plug-pull law outranks write throughput at this volume); store identity fixed at open — one device, one store; multiple handles on one file are legal for reads (WAL), single writer is the usage contract (enforcement not required at v1).

- **API contract (binding for the test session):**
  - **Placement:** folds live in `packages/sync-client` and are always-on — `openStore({ path, identity })` (unchanged signature) creates/maintains the fold tables in the same SQLite file and guarantees, on return, state tables ≡ `refold()` (fold state is durable with the ledger and self-heals on reopen). Apps never register folds (FOLDS.md registration rule, 18 §6).
  - **Canonical replay order (the determinism anchor):** fold state is defined as the fold of the stored event **set** replayed in the total order `key(e) = (global_seq(e) ?? +∞, device_created_at, device_id, lamport_seq)`, lexicographic. Arrival order never matters (01-N1). When every competitor carries a `global_seq`, this is exactly cloud order (01-F34); before assignment the tail key is the deterministic provisional order and every device holding the same set agrees. Incremental maintenance is an implementation choice; the law is equivalence with canonical replay.
  - **New store surface (additive to T-01-03; nothing existing changes shape):**
    - `store.ingest(envelope: EventEnvelopeT, opts?: { global_seq?: int ≥ 0 }): { stored: boolean }` — the branch-stream entry point (mesh/cloud delivery wires in at T-01-05; tests call it directly). Validates via domain `parseEvent` — unknown type/invalid payload throws, nothing persisted (01-F4). `org_id`/`branch_id` must equal the store identity (mismatch throws). An envelope `id` already stored (own or peer) with no `opts.global_seq` → `{ stored: false }`, zero state change (idempotence); a duplicate-id ingest that CARRIES `opts.global_seq` adopts the cloud ordering exactly as `assignGlobalSeq(id, global_seq)` would — idempotent on the same value, divergent value or already-held seq throws, and the call returns `{ stored: false }` (the envelope itself was not newly stored; only the ordering was adopted) — so the LAN-first-then-cloud-catchup path converges (01-F34; adversarial-review finding: silently dropping the carried seq left same-information devices divergent). An own-`device_id` envelope not already stored throws (own events enter only via `append` — loud, 18 §4). Persists to the new `peer_events` table (`id PK, device_id, lamport_seq, envelope; UNIQUE(device_id, lamport_seq)` — a (device, lamport) collision with a different id throws: lamport is gap-free monotonic per device, so a collision is corruption; cloud order lives ONLY in the `global_seq` sidecar — a mirror column on `peer_events` was cut in review as write-only dead data that silently goes stale) and applies folds in the same transaction.
    - `store.append(input)` — unchanged contract from T-01-03, now also applies folds in the same transaction as the insert. Append NEVER fails or blocks for fold reasons (a sale is never blocked, 01-F17): an appended event with an unmet dependency persists to the ledger and parks at the fold layer.
    - `store.assignGlobalSeq(event_id: string, global_seq: int ≥ 0): void` — records cloud ordering for an already-stored own or peer event in a sidecar table (`global_seq` map keyed by event id; envelope rows are never updated, 01-F1). Same value again = idempotent no-op; a different value for the same event, an unknown `event_id`, or a `global_seq` already held by another event throws (protocol corruption — loud, 18 §4). After it returns, state tables ≡ `refold()` (this is how "devices converge to cloud ordering on ack" is observable).
    - `store.openOrders(): OpenOrderRow[]` (sorted by `order_id` asc), `store.kitchenQueue(): KitchenQueueRow[]` (sorted by `order_id` asc), `store.parked(): ParkedRow[]` (sorted by `event_id` asc), `store.refold(): void` (drop + rebuild all fold tables from `events` ∪ `peer_events` + sidecar — the device-side seed of the 20 §4.2 refold-and-diff pattern).
  - **`open_orders` state (SQLite `orders(order_id PK, channel, order_type, table_id, confirmed_at, settled, json_lines)` — exactly FOLDS.md):**
    - `order.created` → row `{ order_id, channel, order_type: payload.order_type ?? NULL, table_id: payload.table_id ?? NULL, confirmed_at: NULL, settled: 0, json_lines: {} }`. A second `order.created` for the same `order_id`: first in canonical order wins, later ones are logged no-ops.
    - `order.confirmed` → `confirmed_at = device_created_at` of the canonically-first confirm; later confirms are logged no-ops.
    - `order.line_added` → line `{ item_id, qty, unit_price_paisa, state: "placed", anomalies: {} }` under `line_id`; duplicate `line_id`: first in canonical order wins. Lines are never removed at v1 (`line_removed` deferred; post-KOT removal is `void.recorded` per 01 §4†).
    - `order.line_state_changed` → for each distinct `line_id` in `line_ids` (duplicates within one event are deduped in the fold loop — an applied event must never self-flag an anomaly; adversarial-review finding), `applyLineState(current, payload.state)`; non-applied results write `anomalies[envelope.id] = "terminal_regression" | "illegal_transition"` on that line (retained + flagged, 01-F35; the ledger row is untouched).
    - `order.table_assigned` → `table_id` = payload of the canonically-**last** assignment (the 01-F34 order-sensitive field: once all competitors carry `global_seq`, highest `global_seq` wins on every device).
    - `payment.recorded` / `payment.refunded` → `settled = 1` iff `billed_effective > 0 ∧ Σ payments.amount_paisa − Σ refunds.amount_paisa == billed_effective`, where `billed_effective = Σ qty × unit_price_paisa` over lines with state ∉ {voided, cancelled}, and a refund attributes to the order of its parent payment. `settled` is derived, recomputed deterministically from the set (v1 approximation of 01-F30 — void/comp/discount value terms are 0 until their event types land; refund-remainder enforcement (01-F29) is hub/cloud quarantine work, 01-F37 — both listed above as constraints-not-closed).
    - `json_lines` and `envelope_json` are canonical JSON (object keys sorted at every depth **by UTF-16 code unit** — pinned so a future non-JS refold-and-diff implementation cannot byte-mismatch, 20 §4.2 — no insignificant whitespace) — determinism assertions compare byte-for-byte.
  - **`kitchen_queue` state (SQLite `queue(order_id PK, confirm_at, channel, age_basis, lines_ready, lines_total)` — exactly FOLDS.md):**
    - A row exists iff a canonically-applied `order.confirmed` exists for the order. `confirm_at = device_created_at` of the canonically-first confirm; `channel` from the order's `order.created` (the fold therefore also consumes `order.created`/`order.line_added` — FOLDS.md's consumes list is read as the trigger list; drift flagged for a FOLDS.md correction).
    - `age_basis = device_created_at` of the canonically-first `kot.printed` for the order, else `confirm_at` (kitchen age runs from ticket print when one exists; doc 03 owns timing refinements).
    - `lines_total` = count of lines with state ∉ {voided, cancelled}; `lines_ready` = count with state ∈ {ready, served, picked_up, delivered} (feeds 03-F19/F24's order-ready-is-a-fold).
    - No row exit at v1 — consumers filter; bump/served-exit semantics are doc 03 work.
  - **Parking (01-F10; SQLite `parked(event_id PK, waiting_for, envelope_json)` — exactly FOLDS.md):**
    - Dependencies come from typed payloads (not raw `refs[]`, which stays advisory at v1): every consumed type except `order.created` requires its `order_id` resolved (an applied `order.created` with that payload id); `order.line_state_changed` additionally requires every `line_id` in `line_ids` resolved (an applied `order.line_added`); `payment.refunded` requires `payload.payment_id` resolved (a stored-and-applied `payment.recorded` **event id**).
    - An event with an unmet dependency is persisted to the ledger normally but parks: one `parked` row, `waiting_for` = the first unmet id (deterministic: order id, then lowest missing line id, then payment event id), and it affects NO state table while parked.
    - Drain: when an applied event resolves an id (`order.created` resolves its order id, `order.line_added` its line id, `payment.recorded` its envelope id), parked rows waiting on that id re-attempt in canonical order; still-unmet dependencies re-park under the new `waiting_for`; cascades (refund → payment → order) run to fixpoint. Duplicate ingest of a parked event's id is a no-op (PK).
    - Nothing is ever dropped: every stored envelope of a consumed type is at all times either applied or parked. Rolling-window parked-age anomaly → fleet health is later doc-15 wiring; `store.parked()` is the v1 observability.
  - **Laws (each is an acceptance test; FR id in the test name per 20 §4.1):**
    1. **Determinism (01-N1):** for any registry-valid generated event set (N devices, per-device gap-free lamport) and ANY two delivery interleavings that respect each device's lamport order (own events via `append`, peers via `ingest`), final `orders`/`queue`/`parked` tables are identical, `json_lines` byte-identical (fast-check property; named generator).
    2. **Commutativity (01-F34):** swapping two adjacent deliveries from different devices yields identical final tables (property).
    3. **Idempotence (01-F34):** re-delivering any already-stored events (same ids, any subset, any order) changes nothing — tables byte-identical before/after (property).
    4. **Refold equivalence (01-F6):** after any delivery sequence, `refold()` reproduces the incrementally-maintained tables exactly (property; the Auditor pattern, 20 §4.2).
    5. **Terminal monotonicity (01-F35, regression guard):** a line in a terminal state receiving a later non-terminal `line_state_changed` keeps its state, gains exactly one `anomalies[event_id] = "terminal_regression"` entry, and the ignored event remains readable in the ledger.
    6. **Parking (01-F10):** child-before-parent delivery → parked row present, state tables unaffected; parent arrival → parked row gone, final state identical to parent-first delivery; the refund→payment→order chain delivered fully reversed drains to fixpoint; no consumed event is ever dropped (applied ∪ parked = stored).
    7. **Tiebreak/convergence (01-F34):** two concurrent `order.table_assigned` from different devices with no `global_seq` resolve identically on every delivery order via the provisional key; after `assignGlobalSeq` on both such that cloud order reverses the provisional winner, `table_id` is the higher-`global_seq` value and tables ≡ `refold()`.
    8. **Durability (extends T-01-03's kill-seed):** after abrupt handle abandon (no `close()`), `openStore` on the same path yields state tables ≡ `refold()` of the surviving ledger — confirmed folds are never ahead of or behind the confirmed ledger.

    5. **Merge (01-F3):** surviving events get `server_received_at = clock.now()` and `global_seq` = next value from the org's counter row, assigned in array order inside **one transaction** with the inserts and the watermark update.
    - `acked_watermark` = highest lamport that is contiguously **persisted** for the device, where persisted = merged into `kernel.events` **or** stored in `kernel.quarantine` — a quarantined event is durably held, so the ack advances over it and the device outbox never wedges on a poisoned event (assumption below; 01-F17 never-blocked spirit). `push_ack { acked_watermark }` is delivered **only after commit** (01-F2 cloud side — law 3).
    - **Fan-out (01-F9 serve half):** after commit, the merged (non-quarantined) events go out as one `event_batch` (envelopes + `global_seq`) to **every** session in the (org, branch) fan-out set, **including the origin** — that is how a device learns its own events' `global_seq` and converges to cloud ordering (01-F34 / T-01-04 `assignGlobalSeq`). Fan-out passes through `sliceFilter(session, events)` — identity at v1, the named 01-F40 seam. Quarantines additionally produce `quarantine_notice { event_id, reason }` to the pushing session in the same `handle()` call (durable redelivery when the origin is offline at notice time is T-01-08).
  - **Catchup (01-F9):** `catchup_request { from_global_seq }` (session required) → `catchup_response { events, complete, next_from }`: the session's **branch** stream, events with `global_seq > from_global_seq` (exclusive cursor — matches `hello.last_global_seq` / `store.setLastGlobalSeq` "last received" semantics), ascending, at most `CATCHUP_PAGE_SIZE = 500` (exported binding constant); `next_from` = the highest `global_seq` in the page (= `from_global_seq` when the page is empty) so the response feeds the next request unchanged; `complete` = no further branch events beyond the page at read time.
  - `ping { t }` → `pong { t }` on the same connection. `purge_command` is never emitted at Wave 0 (revocation is T-01-09+).
  - **Error taxonomy (18 §3/§5):** `ProtocolViolationError`, `AuthRejectedError` extend a `GatewayError` base; quarantine reasons are the closed string set `schema_invalid | org_mismatch | branch_mismatch | device_mismatch | id_content_divergence | lamport_conflict` (invariant-class reasons arrive with T-01-08).
  - **Thin Fastify adapter (in scope, not acceptance-gated):** `src/server.ts` — Fastify + `@fastify/websocket` (18 §5): socket frame → `decodeMessage` → `conn.handle`; sink → `encodeMessage` → `socket.send`; decode/handle errors → log (pino) + socket close. Env (`DATABASE_URL`, `PORT`) via the `packages/config` `defineEnv` factory (crash at boot on invalid env, 18 §5). One boot smoke test at most; real-socket behavior is verified at H-01-D (the first 20 §4.5 reality gate), not here.
- **Laws (each is an acceptance test; FR id in the test name per 20 §4.1):**
  1. **Global sequence (01-F3):** merged events carry `server_received_at` from the injected clock and a strictly monotonic, dense (gap-free) per-org `global_seq` in merge order; two orgs' sequences are independent; a rolled-back merge consumes nothing (counter update rolls back with it).
  2. **Idempotent push (01-F8):** re-pushing any prefix or the whole batch produces the same `acked_watermark`, zero new rows, zero re-fan-out of already-merged events; dedupe is by event `id`; divergent content under a known id quarantines and never mutates the stored row (01-F1).
  3. **Persist-before-ack (01-F2 cloud):** at the moment `push_ack` reaches the sink, every acked event is readable through an **independent** DB connection (the sink itself queries and finds the rows — commit precedes ack, observably).
  4. **Catchup pages never skip and never duplicate (01-F9):** paging from 0 with concurrent merges in flight yields every branch event exactly once in `global_seq` order across pages; `complete`/`next_from` behave per contract; page cap enforced; events of other branches and other orgs never appear (00 §5.4 isolation).
  5. **Fan-out (01-F9/01-F34):** an event pushed by device A reaches every connected same-branch session including A, exactly once, as `event_batch` with `global_seq` present; a session on another branch or org receives nothing; a closed connection is removed from the set (no send after close).
  6. **Quarantine (01-F37):** a registry-invalid event → verbatim `kernel.quarantine` row + `quarantine_notice` on the pushing session + absent from `kernel.events`/fan-out/catchup + **the watermark still advances over it** (the outbox never wedges); re-push of the same bad event is an idempotent no-op; each identity-mismatch reason class is exercised.
  7. **Stop-at-gap (01-F8/01-F3 device-half constraint):** a push whose lamports jump past the expected next stores and acks only the contiguous prefix; a later contiguous re-push completes the sequence; `kernel.events` is lamport-gap-free per device modulo quarantine rows.
  8. **Golden fixtures (20 §2.7):** every fixture under `packages/sync-protocol/src/__acceptance__/fixtures/` (read via repo-relative fs path) decodes with `decodeMessage` and drives the gateway: `hello.json` opens a session, `push.json` merges and acks, `event_batch.json`/`quarantine_notice.json` byte-shape-match what the gateway emits (modulo instance values); every gateway-emitted message round-trips `encodeMessage → decodeMessage` — client and gateway consume the same contract and cannot drift apart silently.

- **Assumptions stated (24 §3b — interpretation chosen / simpler alternative named):**
  1. **Partitioning deferred.** 01 §5 says "partitioned by org + month"; v1 is a plain table with the three indexes. Partitioning now means per-org-per-month DDL automation (a partition-management job) at zero fleet size — speculative machinery (24-F23). The later conversion is a physical reorganization (attach-as-partition / table swap), not a schema break, scheduled with the Wave-1 scale task; recorded for W0-EXIT. Alternative (partition from day one) named and rejected as above.
  2. **Per-org counter row over `BIGSERIAL`.** Chosen: dense per-org sequence, assignment serialized under the row lock held to commit — which is precisely what makes law 4 (catchup never skips) provable; a shared sequence yields per-org gaps *and* the commit-visibility race (seq N+1 visible before N), forcing snapshot guards in catchup. Cost: merges serialize per org — accepted at Wave-0 volume (01-N4 binds at Wave-1 staging). 01-F3 does not require gap-free globally; density is a choice for auditability, not a claimed law of the FR.
  3. **Watermark advances over quarantined events** because they are durably stored (01-F37 storage); the alternative (ack only merged events) wedges the device outbox forever on one poisoned event — the anti-pattern 01-F17's never-blocked principle exists to prevent. Consequence: the Auditor's per-device lamport-gap check must treat quarantine rows as filling their slots — noted for T-01-11.
  4. **Stop-at-gap push processing** over store-beyond-gap: keeps `kernel.events` contiguous per device with a stateless contiguity rule; the device re-pushes from the ack anyway (19 §5), and id-dedupe absorbs the overlap. Store-beyond-gap is harmless but buys nothing and complicates the watermark computation.
  5. **Fan-out includes the origin session** — this is the delivery path by which a device learns its own events' `global_seq` (T-01-04's `assignGlobalSeq`; 01-F34 "converge to cloud ordering on ack"): `push_ack` carries only the watermark, so without origin fan-out a device needs a catchup round-trip to converge. Alternative (exclude origin) named and rejected as slower convergence for zero saved complexity (id-dedupe makes the duplicate delivery free).
  6. **Catchup cursor is exclusive** (`from_global_seq` = highest seq already held): matches `hello.last_global_seq` ("last received") and T-01-03's `store.setLastGlobalSeq` semantics; `next_from` feeds the next request unchanged. PROTOCOL.md doesn't pin inclusivity — flagged for a one-line clarification PR at W0-EXIT (same class as the `resume_from` LAN note from T-01-05).
  7. **Dev-token stub carries org claims.** `hello` has no `org_id` field, and the session must know its org before any push (catchup + fan-out registration are org/branch-scoped) — so the Wave-0 stub token is unsigned base64url JSON claims `{ org_id, branch_id, device_id }` behind the `verifyDeviceToken` seam; T-01-09 replaces the internals with `jose` + registry/revocation, same claims out. Alternative (trust `hello` fields, learn org from the first pushed envelope) rejected: leaves the session org-less for catchup and inverts the 18 §5 server-side-validation posture. The stub is validation of *shape and consistency* only — stated plainly so nobody mistakes it for auth (DEC rows T-01-09/W1).
  8. **Acceptance tests drive the core module surface in-process; transport is a seam.** Real sockets in the acceptance suite would be nondeterministic and duplicate H-01-D's rung — 20 §1 places real-process scenarios in the containerized sim-branch. The Fastify adapter still ships (18 §5 names the stack; H-01-D needs a bootable service) but thin and boot-smoked.
  9. **In-memory session registry; single gateway instance at Wave 0.** Multi-instance fan-out needs a pub/sub backplane (Redis is registry-allowed) — that is Wave-1 scale work ("scales separately from api", 18 §2); building it now is 24-F23 speculative flexibility. Named as a seam, not built.
  10. **No error wire kind:** PROTOCOL.md's message set is closed; protocol violations log + close the connection (WS close code at the adapter). Adding an `error` kind would be a PROTOCOL.md spec-review event — not owed at Wave 0.
  11. **`text` columns for ids**, not `uuid`: the envelope schema types ids as non-empty strings (T-01-01), not format-validated UUIDs; the storage layer must not tighten the wire contract (a foreign-format id would otherwise fail at the DB instead of at validation, bypassing quarantine).
  12. **`envelope jsonb` stored verbatim as received**, cloud-stamped values in their own columns and merged at serve time — the stored payload is exactly what the device signed off on (bytes-level fidelity for the future 01-F5 hash chain), and no row update ever back-fills anything (01-F1).
- **Fix-round amendments (adversarial review; each is binding contract text):**
  1. **In-batch dedupe (blocker):** the step-3 dedupe view includes events merged earlier in the SAME transaction — a repeated id within one push dedupes (identical content → skip and count toward the watermark; divergent → `id_content_divergence`), never reaching a duplicate-PK crash that aborts sibling events and wedges the outbox (verified: re-push of the same batch could never succeed).
  2. **Uniform persisted-slot rule (blocker; DEC-SYNC-005):** EVERY quarantine class fills its lamport slot for the watermark computation — `id_content_divergence` included (the shipped code filled identity/schema classes only; a divergent event beyond the frontier wedged everything after it).
  3. **`storage_reject` joins the closed reason set**, and `kernel.quarantine.envelope` becomes `text` (verbatim JSON string): bytes Postgres cannot faithfully hold in `jsonb` (e.g. ` ` in any string — passes Zod, aborts the insert) quarantine as `storage_reject` instead of aborting the merge; per-event savepoint isolation makes any single event's storage failure quarantine that event and never abort siblings or suppress the ack. `kernel.events` keeps `jsonb` (only storable events reach it, by construction).
  4. **No `push_ack` when nothing is contiguously persisted** for the device (`through < 0` — empty push, or gap-at-start on a fresh device): an ack of 0 would claim slot 0 is held; mirrors the LAN hub's guard (mesh-session sends acks only at `acked ≥ 0`).
  5. **`handle()` serializes per connection** — a frame never begins processing before the previous frame settles (kills the double-`hello` TOCTOU that could register one connection in two orgs' fan-out sets, 00 §5.4; the shipped ws adapter dispatches frames un-awaited, so the race was production-reachable).
  6. `@restos/domain` + `@restos/sync-protocol` are **runtime `dependencies`** (parseEvent/codec sit on production paths; the drafted dev classification would break a pruned deploy). The `packages/config` `exports` entry is ratified as binding (mechanically required; form identical to sibling packages).
  7. **Golden-fixture corrections (20 §2.7 spec-review, planner-approved):** `push.json`'s `order.created` payload gains the registry-required `channel`; `hello.json`'s token becomes the Wave-0 dev-token shape (unsigned base64url claims). Both landed by the gateway's test-owning session; law 8 then drives every fixture verbatim (its in-memory substitutions are removed).
