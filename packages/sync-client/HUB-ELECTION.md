# Hub election state machine (24-F8 artifact for spec 01-F13 — implementation follows in T-01-05)

States: `solo` (no peers) → `follower` (connected to hub) → `candidate` (election) → `hub`.

- **Eligibility:** hub-eligible classes only (01-F39): counter_electron > counter_rn > kitchen. Waiter/manager/rider devices never enter `candidate`.
- **Deterministic election (no consensus protocol needed):** every eligible device broadcasts `ping{class, device_id}` on LAN (mDNS discovery, manual-IP fallback, 01-F12). Winner = highest class rank; tie → lowest `device_id` (lexicographic). Every device computes the same winner locally from the same peer set — election is a pure function of visible peers, re-run on peer-set change.
- **Heartbeat:** hub pings followers every 2 s; follower marks hub lost after 3 missed (6 s) → recompute election from current peers → new hub announces via `hello_ack{hub:true}`. Re-election target < 10 s (01-F13).
- **Split-brain tolerance:** two hubs after a partition is *safe by design* — both relay append-only events; merge on heal is set-union + dedupe (01-F8, 01-F38); the deterministic function converges both sides to one hub once they see each other. No fencing needed because hubs hold no exclusive authority (cloud assigns `global_seq`, 01-F3).
- **Hub duties:** relay `push` branch-wide; serve `catchup` from its full window (01-F14).
- **Cloud uplink: per-device where possible, hub-relayed where not (DEC-SYNC-009, accepted — supersedes DEC-SYNC-004):** a device that has WAN runs its own cloud session. A device that has **no** WAN — the normal case when the client gives internet only to the counter terminal, and deliberately keeps waiters off it — has its events relayed upward by the hub on its behalf. Without this a LAN-only device's events reach the counter but never reach the cloud, silently truncating the cloud log. Relayed events are attested by the hub and carried verbatim (origin `device_id`/`lamport_seq`/payload unchanged, 01-F1); the cloud tracks lamport contiguity per origin device. **Implementation status: NOT YET BUILT** — the shipped code still enforces the superseded no-proxy rule (the gateway quarantines a `device_mismatch`), so LAN-only devices are currently cloud-stranded until the DEC-SYNC-009 task lands.
- **LAN heal has a known limit (DEC-SYNC-006, accepted):** because followers push only their own events and only the hub replays its window, a departed origin's events that reached just one side of a partition cannot cross the heal on the LAN alone — they reconcile via cloud catchup when WAN returns. Accepted for Wave 0; follower third-party relay is the deferred alternative if extended-WAN-outage churn demands it. T-01-06 X7 demonstrates the cloud path closes it.
- **Cold start:** single eligible device → `solo` (acts as hub for later joiners). Scoped classes connect to whoever is hub and never serve peers (01-F39).
